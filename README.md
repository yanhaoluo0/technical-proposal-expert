# Technical Proposal Expert

> 专为政府和企业软件项目技术标书撰写而设计的 AI 助手，将资深技术标书专家的经验与规范转化为可执行的写作能力。

## 🚀 快速开始

### 第一步：启动对话
直接开始对话，系统会自动：
1. 创建 `plan.md` 文档（项目约定的信息文档）
2. 进入**大文本编辑模式**（默认开启）
3. 开始引导您完成技术方案确认

### 第二步：确认技术方案
系统会询问您以下信息：
- 技术架构选型
- 主要功能模块划分
- 业务场景描述
- 系统定位

### 第三步：确认需求清单
系统会基于技术方案整理功能点列表，以表格形式展示，您只需确认每个功能点。

### 第四步：收集指标性内容
系统会询问是否有性能指标、安全指标、合规要求等必须在设计中体现的内容。

### 第五步：开始撰写
确认计划后，系统会请求您指定先写哪个功能模块，然后按计划逐步撰写。

## ✨ 核心特性

### 1. 零分点原则
严格遵循公文写作规范，禁止在正文中使用任何形式的分点符号（`-`、`*`、`1.`、`•`），所有内容均以叙述性段落呈现，确保文档的专业性和正式性。

### 2. 标准公文结构
采用标准公文层级结构（`一、` -> `(一)` -> `1.` -> `(1)`），段落长度控制在 150-300 字之间，避免碎片化表达。

### 3. 需求响应机制
每个功能点或模块的描述都明确对应招标文件中的具体需求条款，包含**功能描述**、**实现流程**和**业务流程设计**三个维度，确保内容直接响应招标要求。

### 4. 计划驱动执行（大文本编辑模式）
- **自动创建 `plan.md`**：系统会自动创建项目计划文档，记录技术方案、需求清单、指标性内容和撰写进度
- **分步执行**：每完成 1000 字内容即暂停等待用户确认，确保质量可控
- **实时更新进度**：撰写完成后自动更新 `plan.md`，标记已完成的功能点（格式：`[x] 功能模块/功能点`）

### 5. 小标题分解思维框架
系统在撰写每个功能点前，会按照以下优先级思考并确定小标题结构：
1. **功能定位与价值**（最重要）：核心定位、业务价值、应用场景
2. **技术架构设计**：技术选型、核心技术点、架构图说明
3. **功能实现细节**：核心功能点、技术实现路径、关键环节
4. **业务流程设计**：完整业务流程、业务规则、流转逻辑
5. **性能与保障**：性能指标、可靠性保障（安全作为基础保障，融入各维度）
6. **部署与运维**：部署架构、监控体系、运维策略
7. **创新与优势**（可选）：技术亮点、竞争优势

每个功能点至少包含 2 个小标题，确保内容全面且符合标书专家视角。

### 6. 专业词汇库
内置领域专业词汇库，包含架构类、管理类、价值类等专业术语，同时提供禁词表，避免 AI 惯用语，提升文档的公文质感。

## 📋 工作流程详解

### 大文本编辑模式（默认开启）

#### Phase 1: 技术方案明确
- 系统进行简单询问，明确标书的技术方案等基本信息
- 将您的回答整理为技术方案简述，写入 `plan.md`

#### Phase 2: 需求确认
- 系统基于技术方案整理功能点列表
- 以表格形式展示需求清单（包含技术方案简述）
- 您只需确认每个功能点（是/否）

#### Phase 3: 指标性内容收集
- 系统询问是否有相关指标性质的内容（性能指标、安全指标、合规要求等）
- 如有，系统会明确其关联的功能点，并添加到 `plan.md`

#### Phase 4: 计划完成与任务选择
- 系统展示完整的 `plan.md` 计划文档
- 请求您自行判断先写什么部分（系统会提醒注意上下文限制）

#### Phase 5: 迭代执行（分段输出）
- 根据您指定的功能模块，锁定当前需要完成的功能单元
- **小标题分解**：按照思维框架确定小标题结构
- **执行写作**：按照零分点原则撰写，每 1000 字暂停等待确认
- **标记进度**：完成后自动更新 `plan.md`，标记为 `[x]`

#### Phase 6: 循环执行
- 等待您确认后，继续下一个功能模块，直到计划全部完成

### 简单问答模式（可选）

如果明确表示不开启大文本编辑模式，系统会：
- 不创建 `plan.md` 文件
- 采用简单的问答模式，直接响应撰写需求
- 内容约束保持一致（零分点原则、需求响应原则等）

## 📁 plan.md 文件说明

`plan.md` 是本次项目约定的信息文档，包含以下章节：

### 一、技术方案简述
记录项目的技术架构选型、主要功能模块、业务场景等关键信息。

### 二、需求清单
以表格形式展示功能点列表，包含：
- 序号
- 功能模块/功能点
- 技术方案简述
- 用户确认状态

### 三、指标性内容
记录性能指标、安全指标、合规要求等必须在设计中体现的内容，并标注关联的功能模块。

### 四、撰写进度
跟踪每个功能点的撰写状态，格式为：`[ ] 功能模块/功能点`（未开始）或 `[x] 功能模块/功能点`（已完成）。

**注意**：请勿手动删除 `plan.md` 文件，系统会持续更新该文件以跟踪项目进度。

## 📊 示例对比

### ❌ 不符合规范的输出

```
3.1 安全性设计
* 我们使用了HTTPS协议。
* 数据库有备份。
* 防火墙可以防止攻击。
* 只有管理员能删除数据。
```

### ✅ 符合规范的输出

```
**(一) 安全防护体系设计**

针对本项目对数据隐私性和系统连续性的严苛要求，安全架构设计遵循"纵深防御"理念。在传输层，系统强制启用全链路 SSL/TLS 加密通道，确保数据在公网传输过程中的不可窃听性。数据存储层面，不仅实施了每日增量与每周全量的自动化备份策略，更引入了异地容灾机制以应对极端物理灾害。在网络边界防护上，通过部署下一代应用防火墙（WAF）与入侵防御系统（IPS），构建起针对 SQL 注入、XSS 跨站攻击的主动防御屏障。此外，权限体系严格遵循 RBAC 模型，确保敏感操作权限仅授予经过授权的核心管理人员，并配合审计日志实现操作的可追溯性。
```

- **语言标准**：简体中文（中国大陆政府/企业标准）
- **文档标准**：GB/T 8567-2006（软件文档编写标准）

## ⚠️ 注意事项

1. **分段确认**：撰写长文档时，每完成 1000 字内容会暂停等待确认，请及时反馈
2. **需求对应**：建议提供招标文件或需求文档，以便系统准确响应要求
3. **计划文件**：系统会自动创建 `plan.md` 文件跟踪进度，请勿手动删除
4. **格式要求**：输出内容严格遵循零分点原则，如需列表格式请明确说明
5. **上下文限制**：选择撰写顺序时，建议按照逻辑依赖关系（如先写基础架构，再写业务功能）
